Каждая из этих аудиторий обладает уникальными свойствами, а поэтому важна для большинства проектов.<br/>
<br/>
Данная статья задумывалась как открытая попытка разобраться, на чем стоит писать мультиплатформенное desktop-приложение. Приглашаю высказать свое мнение людей с опытом создания таких приложений.<br/>
<br/>
Изначально определим критерии, по которым будем оценивать различные платформы. В первую очередь — это удобство пользователя. Уверен, что вы замечали общие черты у приложений, написанных под разные платформы. Во вторую — это интересы проекта. Моя задача — построить успешный бизнес, а не играться с различными инструментами.<br/>
<br/>
Обозначим рамки исследования. Мое приложение — небольшая утилита для пользователя-«чайника», которая качает файлы из интернета: минимум GUI, небольшой набор функциональности, использование внешних С++ библиотек.<br/>
<br/>
Ну что, начнем. Какие есть варианты? Я рассмотрю Java, C#, C++, Python. Буду рад, если вы расскажите о других альтернативах. <a name="habracut"></a><br/>
<br/>
<h2>Java</h2>Данный язык/среда изначально задумывались как нечто мультиплатформенное. На Java написано большое количество приложений, крупные проекты вроде Eclipse используют именно этот фреймворк.<br/>
<br/>
Большой минус Java с точки зрения пользователя — необходимость устанавливать фреймворк. Это безусловно сложнее установки Flash player'а и часто становится pain in the ass. И размер инсталлятора (я качал 80+ мб), и постоянная путаница в названиях (JRE, J2SE, JDK, JVM, ...) не играют на руку разработчикам приложений под Java.<br/>
<br/>
<a href="http://en.wikipedia.org/wiki/Learning_curve">Learning curve</a> имеет критическое значение для любого проекта, который ставит на вирусное продвижение. Если 30% ваших пользователей не смогут поставить Java-машину (лень качать, неудобный сайт, отложили «на потом», ...) — у ваших конкурентов будет фора в 30%. Эта огромная доля пользователей, потерять которых мой проект себе позволить не может.<br/>
<br/>
Все приложения, которыми мне доводилось пользоваться, не использовали «родной» интерфейс Win32. Не знаю, чем руководствовались разработчики, но с точки зрения конечного пользователя это выглядит очень не симпатично.<br/>
<br/>
<strong>Примеры приложений</strong>: Eclipse, ZDE, клиент для Gnutella Limewire.<br/>
<strong>Плюсы</strong>: мультиплатформенность, большое количество кадров, развитость фреймворка.<br/>
<strong>Минусы</strong>: необходимость установки фреймворка, кривость GUI, низкая производительность.<br/>
<br/>
<h2>C#</h2>Язык C# и платформа .NET вышли из под крыла компании Microsoft, которая никогда не отличалась уважением к мультиплатформенности. Данный язык был тепло принят разработчиками и энтузиасты <a href="http://en.wikipedia.org/wiki/Mono_%28software%29">проекта Mono</a> даже сделали аналог фреймворка под Linux/Unix/Mac Os X.<br/>
<br/>
Аналогично с Java, приложения на C# имеют большой минус — необходимость устанавливать фреймворк. Я сам отказался от установки несколько приложений, которые требовали этого фреймворка. Менее продвинутые пользователи будут ещё более требовательными.<br/>
<br/>
В остальном — сплошные плюсы, на мой взгляд.<br/>
<br/>
<strong>Примеры приложений</strong>:?<br/>
<strong>Плюсы</strong>: мультиплатформенность, большое количество кадров, хорошая производительность, развитость фреймворка.<br/>
<strong>Минусы</strong>: необходимость установки фреймворка.<br/>
<br/>
<h2>C++</h2>Старичок дотянул до наших дней и замечательно себя чувствует. Много приложений под платформы Linux и Windows до сих пор пишутся на этом языке.<br/>
<br/>
Программы, написанные на C++, являются примером для других по размеру дистрибутива и экономному использованию системных ресурсов (процессор, память). Тем не менее у разработчиков есть <a href="http://en.wikipedia.org/wiki/Criticism_of_C%2B%2B">масса претензий</a> к С++. По моему мнению, язык является «устаревшим» и его популярность в дальнейшем будет снижаться, что подтверждается <a href="http://www.tiobe.com/index.php/content/paperinfo/tpci/index.html">индексом TIOBE</a>.<br/>
<br/>
С точки зрения развития проекта, по сравнению с динамическими интерпретируемыми языками (вроде Ruby и Python), разработка на данном языке может иметь менее высокую скорость и более высокие издержки изменения проекта. Для стартапа, которому не столь важна производительность приложения, это может стать существенным минусом.<br/>
<br/>
<strong>Примеры</strong>: Firefox,?..<br/>
<strong>Плюсы</strong>: отличная производительность, большое количество кадров, большое количество библиотек.<br/>
<strong>Минусы</strong>: невысокая скорость разработки.<br/>
<br/>
<h2>Python</h2>Взглянуть на Python в качестве платформы для desktop-приложений меня заставила программа <a href="http://musicbrainz.org/doc/PicardTagger">MusicBrainz Picard</a>. Несмотря на свою скриптовую сущность, Python легко собирается в один exe-файл, не требуя от пользователя установки дополнительных компонентов.<br/>
<br/>
В случае разработки небольшого приложения, интерпретируемые языки вроде Python будут большим плюсом. Легкость написания и высокая скорость изменения приложения пригодится любому стартапу.<br/>
<br/>
Огромным минусом различных «модных» технологий является их низкое распространение, а значит серьезные проблемы в поиске квалифицированных кадров. Ситуация с поиском программистов итак плачевная, а если ограничиться узким языком — можно вообще никого не найти. С другой стороны, храбрость перейти на новый язык имеют наиболее прогрессивные разработчики. Может получиться так, что выбрав «перспективный» язык, мы сразу отсечем миллионы середнячков, оставив себе выбор из нескольких перспективных разработчиков.<br/>
<br/>
<strong>Пример</strong>: MusicBrainz Picard, оригинальный BitTorrent.<br/>
<strong>Плюсы</strong>: высокая скорость разработки и изменений, хорошая интеграция с библиотеками на С и С++.<br/>
<strong>Минусы</strong>: мало кадров, низкая производительность.<br/>
<br/>
<h2>Выводы</h2>К сожалению, любая из вышеперечисленных платформ имеет свои плюсы и минусы, однозначного решения найти не удалось. Выбор одной из них сегодня принесет преимущества и недостатки, влияние которых на проект мы увидим только завтра.<br/>
<br/>
Я пока выбираю между С++ и Python. Первый является «надежным» решением с известными недостатками. Второй является «рискованным», но интересным и перспективным. Надеюсь, ваши отзывы помогут мне сделать окончательный выбор. Какую платформу выбрали бы вы на моем месте?<br/>
<br/>
PS.
